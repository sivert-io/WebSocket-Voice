# ü§ù Contributing to Gryt

We welcome contributions to Gryt! This guide will help you get started with contributing to our voice chat platform.

## üöÄ Getting Started

### Prerequisites

- **Node.js** 18+ and **Bun** (for client/server development)
- **Go** 1.21+ (for SFU development)
- **Git** for version control
- **Docker** (optional, for containerized development)

### Development Setup

1. **Fork the repository**
   ```bash
   # Fork on GitHub, then clone your fork
   git clone https://github.com/YOUR_USERNAME/WebSocket-Voice.git
   cd WebSocket-Voice/webrtc
   ```

2. **Set up development environment**
   ```bash
   # Install dependencies for all components
   cd client && bun install
   cd ../server && bun install
   cd ../sfu-v2 && go mod download
   cd ../docs && npm install
   ```

3. **Start development servers**
   ```bash
   # Use the development script
   chmod +x start_dev.sh
   ./start_dev.sh
   ```

## üîß Development Guidelines

### Code Style

#### TypeScript/JavaScript

- Use **TypeScript** for all new code
- Follow **ESLint** and **Prettier** configurations
- Use **meaningful variable names**
- Add **JSDoc comments** for public APIs
- Write **comprehensive tests**

```typescript
/**
 * Manages microphone input and audio processing
 * @param shouldAccess - Whether to request microphone access
 * @returns Audio processing state and controls
 */
const useMicrophone = (shouldAccess: boolean): MicrophoneState => {
  // Implementation
};
```

#### Go

- Follow **gofmt** formatting
- Use **meaningful variable names**
- Add **godoc comments** for public functions
- Write **comprehensive tests**
- Handle **errors properly**

```go
// ProcessAudio applies audio processing pipeline to the given buffer
// Returns the processed audio buffer or an error if processing fails
func (p *AudioProcessor) ProcessAudio(buffer AudioBuffer) (AudioBuffer, error) {
    // Implementation
}
```

### Testing

#### Client Testing

```bash
# Run all tests
cd client
bun test

# Run specific test suites
bun test audio
bun test webrtc
bun test components

# Coverage report
bun test --coverage
```

#### Server Testing

```bash
# Run all tests
cd server
bun test

# Run specific test suites
bun test websocket
bun test signaling
bun test rooms

# Integration tests
bun test:integration
```

#### SFU Testing

```bash
# Run all tests
cd sfu-v2
go test ./...

# Run with coverage
go test -cover ./...

# Benchmark tests
go test -bench=. ./...
```

## üîÑ Pull Request Process

### Before Submitting

1. **Ensure tests pass**
   ```bash
   # Run all tests
   bun test
   go test ./...
   ```

2. **Check code style**
   ```bash
   # Lint and format
   bun run lint
   bun run format
   go fmt ./...
   ```

3. **Update documentation**
   - Add/update JSDoc/godoc comments
   - Update README files
   - Add usage examples

4. **Test your changes**
   - Test in different browsers
   - Test with different configurations
   - Test edge cases

### Pull Request Template

```markdown
## Description
Brief description of changes

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Breaking change
- [ ] Documentation update

## Testing
- [ ] Tests pass locally
- [ ] Added tests for new functionality
- [ ] Tested in multiple browsers
- [ ] Tested with different configurations

## Checklist
- [ ] Code follows style guidelines
- [ ] Self-review completed
- [ ] Documentation updated
- [ ] No breaking changes (or marked as such)
```

### Review Process

1. **Automated checks** must pass
2. **Code review** by maintainers
3. **Testing** by maintainers
4. **Approval** and merge
